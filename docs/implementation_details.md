# Implementation Details: Veeva CRM As-Is Solution Function Generator

This document outlines the technical implementation details of the LangGraph-based Solution Function Generator. The project translates technical Salesforce Component Groups into business-oriented Solution Functions to assist with Vault CRM migrations.

## 1. Project Structure

The codebase is organized as follows:

```text
oca_solution_function_agent/
├── .env                     # API keys and Vault credentials (GEMINI_API_KEY, VAULT_*)
├── main.py                  # Orchestrator script
├── requirements.txt         # Project dependencies
├── src/
│   ├── graph.py             # LangGraph StateGraph definition and routing
│   ├── state/
│   │   └── schema.py        # TypedDict definitions for DomainState and SolutionFunction
│   ├── vector_store.py      # InMemoryVectorStore and similarity search logic
│   ├── tools/
│   │   ├── vault_connector.py # Base Vault API client
│   │   └── vault_tools.py   # Ingestion and clustering logic
│   └── nodes/
│       ├── synthesizer.py   # LLM Builder agent node
│       ├── validator.py     # LLM Critic agent node
│       ├── hitl.py          # Human-in-the-loop interruption node
│       └── vault.py         # Output to Vault and Vector Store sync
```

## 2. Core Technologies

*   **Framework:** LangGraph (v3) / LangChain
*   **LLM:** Google Gemini 3.1 Pro Preview (`gemini-3.1-pro-preview`)
*   **Embeddings:** Google Gemini Embeddings (`models/gemini-embedding-001`)
*   **Vector Store:** LangChain `InMemoryVectorStore`
*   **Data Validation:** Pydantic (for LLM structured output)
*   **Package Management:** `uv`

## 3. Data Ingestion & Clustering (`src/tools/vault_tools.py`)

The entry point for data processing uses `get_component_groups()`.
1.  **Extraction:** Queries Veeva Vault (`oca_component_group__c`) for component groups, complexities, descriptions, and related objects (`component_group_object_joins__cr`).
2.  **Clustering:** Utilizes a Depth-First Search (DFS) algorithm to group component groups into "Candidate Domains" based on overlapping object usage (Relational Proximity).
3.  **Orphans:** Identifies components without object relationships and appends them to candidate domains to ensure no component is left behind.

## 4. State Management (`src/state/schema.py`)

Execution state is passed between LangGraph nodes using a `TypedDict` (`DomainState`).

*   `candidate_domain`: List of input `ComponentGroup` dictionaries.
*   `proposed_functions`: List of `SolutionFunction` dictionaries generated by the LLM.
*   `validation_feedback`: String containing critique from the Validator.
*   `is_valid`: Boolean flag indicating if the proposal passed the rubric.
*   `retry_count`: Annotated integer using `operator.add` to track Builder-Critic loops.

## 5. Agent Nodes (`src/nodes/`)

### 5.1 Synthesizer Agent (`synthesizer.py`)
*   **Role:** Builder.
*   **Implementation:** Uses `ChatGoogleGenerativeAI` with `.with_structured_output(SynthesizerOutput)` to guarantee a Pydantic schema return.
*   **Logic:** Analyzes the Candidate Domain and proposes abstract, business-focused Solution Functions. It consumes any `validation_feedback` from previous loops to correct mistakes.

### 5.2 Validator Agent (`validator.py`)
*   **Role:** Critic.
*   **Implementation:** Evaluates the proposal against a strict Tripartite Rubric:
    1.  **No Orphans:** All components mapped.
    2.  **Business Intent:** No technical jargon (Apex, SOQL, etc.).
    3.  **Registry Overlap:** Before calling the LLM, the node performs a semantic similarity search (`search_similar_functions`) against the `InMemoryVectorStore`. The closest match is injected into the prompt. The LLM rejects the proposal if semantic overlap is >80%.
*   **Output:** Sets `is_valid` and provides detailed `validation_feedback`.

### 5.3 Human-in-the-Loop (`hitl.py`)
*   **Implementation:** Uses LangGraph v3's `interrupt()` primitive.
*   **Logic:** If the `retry_count` reaches 3, the graph pauses execution. It surfaces a payload containing the domain, the last proposal, and the last critique. The orchestrator can later resume the graph via `Command(resume={...})`.

### 5.4 Vault Writer (`vault.py`)
*   **Role:** Finalizer.
*   **Implementation:** Mocks the creation of `oca_solution_function__c` records.
*   **Registry Sync:** Calls `add_solution_function_to_store` to insert the approved function's business description and ID into the local Vector Store, ensuring subsequent domains are checked against this new baseline.

## 6. Graph Orchestration (`src/graph.py` & `main.py`)

*   **Graph Definition (`graph.py`):** Wires the nodes. The `route_validation` edge directs flow back to the synthesizer (retry), to the HITL node (max retries), or to the Vault writer (approved).
*   **Checkpointer:** Uses `MemorySaver()` to persist the graph state, which is required for the `interrupt()` functionality to work correctly.
*   **Main Loop (`main.py`):** 
    1. Fetches and sorts Candidate Domains by total complexity (descending).
    2. Iterates over each domain, assigning a unique `thread_id` to `config`.
    3. Uses `app.stream()` to invoke the graph, providing real-time console feedback on node execution and validation status.
