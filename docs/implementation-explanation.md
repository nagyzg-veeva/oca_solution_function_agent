# Implementation Explanation: LangGraph Builder-Critic Architecture

Based on the architecture we designed and the latest LangGraph framework features (specifically LangGraph v3), this document explains how the application is conceptually implemented in Python.

The application revolves around a **StateGraph**, which maintains a strictly typed state dictionary as it moves between different "Node" functions (our agents).

## 1. The State Schema
First, we define the `State` using Python's `TypedDict`. This state is passed between the nodes. The reducer logic (like `operator.add` or simply replacing values) determines how nodes update the state.

```python
from typing import TypedDict, List, Annotated
import operator

class SolutionFunction(TypedDict):
    name: str
    business_description: str
    primary_objects: List[str]
    component_groups: List[str]
    complexity_score: str

class DomainState(TypedDict):
    # Input data from the Clusterer
    candidate_domain: dict 
    
    # Generated by the Synthesizer
    proposed_functions: List[SolutionFunction] 
    
    # Populated by the Validator
    validation_feedback: str 
    is_valid: bool 
    
    # Loop control
    retry_count: Annotated[int, operator.add] 
```

## 2. The Nodes (Agents & Tools)
Each step of our Builder-Critic loop is a standard Python function that takes the `State`, runs its logic (LLM calls or Python code), and returns a dictionary to update the State.

1. **`synthesizer_node(state)`**: 
   * Uses an LLM (e.g., `ChatOpenAI(model="gpt-4o")`) bound to structured output (Pydantic). 
   * It takes `state["candidate_domain"]` and any existing `state["validation_feedback"]`.
   * It returns `{"proposed_functions": new_proposals, "retry_count": 1}`.
2. **`validator_node(state)`**:
   * Uses an LLM to evaluate the proposals against the Tripartite Rubric.
   * Performs an embedded similarity search against our local FAISS/ChromaDB vector store.
   * If it passes, it returns `{"is_valid": True, "validation_feedback": "Approved."}`.
   * If it fails, it returns `{"is_valid": False, "validation_feedback": "Rejected: Description contains technical jargon 'Apex'."}`.
3. **`hitl_node(state)`**:
   * Uses LangGraph's new `interrupt()` function to pause the graph.
4. **`write_to_vault_node(state)`**:
   * Uses the Vault Connector to write the final `proposed_functions` into `oca_solution_function__c`.
   * Adds the new functions to the local Vector Store registry.

## 3. Control Flow & Routing (Edges)
We wire the nodes together using standard edges and **conditional edges** for the loop.

```python
from langgraph.graph import StateGraph, START, END

# Define the routing logic
def route_validation(state: DomainState):
    if state["is_valid"]:
        return "write_to_vault"
    elif state["retry_count"] >= 3:
        return "hitl_review" # Max retries hit, trigger human fallback
    else:
        return "synthesize" # Loop back to the builder

# Build the Graph
builder = StateGraph(DomainState)

builder.add_node("synthesize", synthesizer_node)
builder.add_node("validate", validator_node)
builder.add_node("hitl_review", hitl_node)
builder.add_node("write_to_vault", write_to_vault_node)

builder.add_edge(START, "synthesize")
builder.add_edge("synthesize", "validate")

# The conditional edge dictates the loop
builder.add_conditional_edges("validate", route_validation)

builder.add_edge("hitl_review", "write_to_vault")
builder.add_edge("write_to_vault", END)
```

## 4. Human-in-the-Loop via `interrupt()`
LangGraph v3 introduced a native `interrupt()` function. When the graph hits the `hitl_node` because of a disagreement loop, the application behaves like this:

```python
from langgraph.types import interrupt, Command

def hitl_node(state: DomainState):
    # This pauses the entire execution and surfaces the conflict payload
    human_decision = interrupt({
        "reason": "Max retries exceeded.",
        "domain": state["candidate_domain"],
        "last_proposal": state["proposed_functions"],
        "last_critique": state["validation_feedback"]
    })
    
    # When resumed, the human's manual corrections are returned
    return {
        "proposed_functions": human_decision["corrected_functions"],
        "is_valid": True 
    }
```

To run this, we compile the graph with a `checkpointer` (e.g., `MemorySaver` or Postgres), which persists the state securely to disk or memory when paused:
```python
from langgraph.checkpoint.memory import MemorySaver

memory = MemorySaver()
app = builder.compile(checkpointer=memory)
```

When a human reviews the conflict and decides on the final Solution Functions, they "resume" the specific thread by passing a `Command`:
```python
# The human resolves the conflict and resumes the graph
app.invoke(
    Command(resume={"corrected_functions": [...]}), 
    config={"configurable": {"thread_id": "domain_42"}}
)
```

## Summary of Execution
The orchestrator script pulls a Candidate Domain from Vault, creates a `thread_id` for it, and calls `app.invoke()`. 
* Most domains will quickly loop between `synthesize` and `validate`, pass validation, go to `write_to_vault`, and end.
* If a domain gets stuck, it halts at `interrupt()`. The orchestrator simply moves on to process the next domain in the queue using a *new* `thread_id`. The human architect can query LangGraph later for all paused threads, review the data, and unpause them.
